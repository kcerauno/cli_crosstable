# コードレビューと改善提案

対象: `crosstable.py` / `tests/test_crosstable.py` (2026-07-19 時点、コミット `9f862b4`)

このドキュメントは評価と提案のみで、コードの変更は行っていない。
「気になる点」に挙げた挙動はすべて実際にコマンドを実行して確認済み。

## 総評

「シンプルなアプリを作る」という当初の要件に対して、適切な規模に収まっている。
単一ファイル・依存は rich のみ・関数4つ+main という構成は、この用途なら正解で、
クラス化や設定ファイル対応などをしなかった判断は良い。
パース→集計→整形→描画が関数単位で分離されているため、テストが書きやすい形になっており、
実際に単体テストとCLI結合テストの両方が揃っている(25件)のはこの規模のツールとしては手厚い。

一方で、**正常系の仕様は固まっているが、異常系の入力に対してはまだ「生のトレースバック」が
そのまま出る**段階にある。空入力のエラー処理は丁寧に作り込まれているのに対し、
壊れた行・不正な数値・存在しないファイルでは argparse のヘルプすら出ない。
SSHログに保存する運用を考えると、ここが実用上いちばん先に直す価値のある部分。

## 良い点

- 関数分離が適切で、`build_crosstab` が I/O を持たない純粋なロジックになっている
- 空入力・端末での引数なし起動を明示的にエラー扱いし、stderr にエラー+使い方を出す設計
- `--ascii` の目的(あいまい幅文字によるログ崩れ対策)が明確で、罫線とマークの両方を対象にしている
- テストが「構造整合性(全行の幅が揃う)」まで見ており、レイアウト崩れの回帰を検出できる
- pty を使って isatty 分岐まで検証している(この規模のツールで省略されがちな部分)
- ソート表示・折り返し表示など、後から追加した仕様がすべてテストで固定されている

## 気になる点(検証済みの実挙動)

### 1. 壊れた入力で生のトレースバックが出る 【優先度: 高】

```
$ printf 'SrvAlice,1\nSrvBob\n' | python crosstable.py
IndexError: list index out of range          # 2列未満の行

$ printf 'SrvAlice,1,abc\n' | python crosstable.py -n
ValueError: could not convert string to float: 'abc'   # 3列目が非数値

$ python crosstable.py nofile.csv
FileNotFoundError: ...                        # ファイルなし
```

いずれもユーザー起因の入力ミスであり、「何行目の何が悪いか」を短く伝えて終了すべき。
既存の `fail()` がまさにその受け皿なので、拡張は自然にできる。

### 2. float 集計による誤差と精度 【優先度: 高】

```
$ printf 'A,r,0.1\nA,r,0.2\n' | python crosstable.py -n
│ r │ 0.30000000000000004 │
```

`values` が `defaultdict(float)` のため、小数の合算で誤差がそのまま表に出る。
また 2^53 を超える整数値も精度を失う。集計値の表なので、この表示は信頼性を疑わせる。

### 3. 数値モードで「実測値 0」と「データなし」が区別できない 【優先度: 中】

サンプル `numeric.csv` には `SrvCharlie,98783914,0`(実測ゼロ)があるが、
表の上ではデータが存在しないセルの `0` と見分けがつかない。
監視系のデータでは「観測してゼロだった」と「観測していない」は意味が違うことが多い。

### 4. 行名・列名のソートが辞書順のみ 【優先度: 中】

```
$ printf 'A,100\nA,50\nA,9\n' | python crosstable.py
│ 100 │ ...   # 数値としては 9 < 50 < 100 の順が自然
│ 50  │
│ 9   │
```

現サンプルは全行名が同じ桁数のため露見しないが、桁数が混在すると不自然になる。

### 5. 細かい点 【優先度: 低】

- `present` が `defaultdict(bool)` だが、実態は集合。`set` の方が意図が伝わる
- 引用符付きフィールド(`"#foo",1`)がコメント扱いされる(csvモジュールで引用符を外した後に `#` 判定しているため)。実害はまず無いが仕様として曖昧
- `-d '\t'` はリテラルの `\t` 2文字として渡り、タブ区切りにならない(シェルで `$'\t'` と書けば回避可能だが、ヘルプに記載がない)
- 出力先が非端末のとき rich は幅80で描画する。列が多いとリダイレクト先でも省略が起きる(`--col-width` で回避可能だが、全体幅を指定する `--width` があるとログ運用では直接的)

## 改善提案(優先度順)

### A. 入力エラーの整形(上記1に対応)

`parse_rows` で列数を検証し、`float()` 変換もここで行い、
失敗時は `fail(parser, f"{行番号}行目: ...")` 形式で報告する。
csv.reader は `reader.line_num` を持つので行番号は無コストで取れる。
ファイルオープンも try/except で包み、`エラー: ファイルが開けません: nofile.csv` とする。
テストは既存の `CliErrorHandlingTest` に3ケース追加するだけでよい。

### B. 集計の精度改善(上記2に対応)

選択肢は2つ。

1. **`Decimal` で集計する**(推奨): `float(fields[2])` → `Decimal(fields[2])` に変える。
   入力文字列をそのまま十進で扱うため誤差が出ず、`format_number` の
   「整数なら小数点を出さない」ロジックもほぼそのまま使える
2. 表示時に丸める(`round(x, 10)` など): 実装は1行だが、閾値の恣意性が残る

### C. 「データなし」セルの表示分離(上記3に対応)

数値モードで、データが存在しないセルを `0` ではなく `−`(ascii時 `.`)にする。
`present` の情報は既に持っているので、`render_table` の1分岐で済む。
既存の挙動を変えたくない場合は `--empty <文字列>` オプションとして追加する手もあるが、
オプションを増やすより既定動作を変える方がこのツールの思想(シンプル)に合う。
※ 既定挙動の変更になるため、採用するかは要判断。

### D. 自然順ソート(上記4に対応)

`sorted(key=...)` のキー関数を「数字列は数値として比較」する形にする(10行程度)。
`SrvAlice2` と `SrvAlice10` のような混在名も自然な順になる。
外部ライブラリ(natsort)を入れるほどではない。

### E. 開発基盤(任意)

- **GitHub Actions**: push 時に `python -m unittest discover -s tests` を回すだけの
  ワークフロー(10行)。リポジトリを公開した以上、テストが回る印は価値がある
- **pyproject.toml + エントリポイント**: `pip install .` で `crosstable` コマンドになる。
  配布する予定が出てきたら。単一ファイルで `python crosstable.py` する運用のままなら不要
- 型ヒント: 関数境界(引数・戻り値)のみ付ける。この規模なら mypy 導入までは不要

## 提案の採用順序(もし全部やるなら)

1. A(入力エラー整形)— 実用上の効果が最も大きく、既存設計の延長で書ける
2. B(Decimal 集計)— 表の信頼性に直結し、変更範囲が小さい
3. D(自然順ソート)— 小さい変更で表示品質が上がる
4. C(データなし表示)— 既定挙動の変更を含むため、仕様判断を先に
5. E(CI など)— 上記と独立にいつでも
